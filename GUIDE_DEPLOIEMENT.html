1. Prérequis (à installer une seule fois)

  • Node.js (version 18 ou supérieure) — https://nodejs.org
  • Git — https://git-scm.com
  • VSCode — https://code.visualstudio.com
  • Un compte GitHub avec accès au dépôt sandrinepichla/amsp-website

2. Ouvrir le projet dans VSCode

  1. Ouvrir VSCode
  2. Fichier → Ouvrir le dossier → sélectionner C:\Dev\AMSP\amsp-project\amsp-website
  3. Ouvrir le terminal intégré : Ctrl + ù (ou menu Terminal → Nouveau terminal)

3. Lancer le serveur de développement (pour tester en local)

Dans le terminal, taper la commande suivante et appuyer sur Entrée :

npm run dev

Puis ouvrir dans le navigateur : http://localhost:8080

💡 Le serveur se met à jour automatiquement à chaque modification de fichier — pas besoin de le relancer.
Pour l'arrêter : appuyer sur Ctrl + C dans le terminal.

4. Modifier le code

  • Modifier les fichiers dans le dossier src/
  • Le navigateur se recharge automatiquement
  • Vérifier le résultat sur http://localhost:8080 avant de déployer

5. Déployer sur le site en ligne (GitHub Pages)

Le site est hébergé sur GitHub Pages à l'adresse :
https://sandrinepichla.github.io/amsp-website

Étape 1 — Enregistrer les modifications dans Git

Dans le terminal VSCode, taper ces deux commandes l'une après l'autre :

git add .
git commit -m "Description de ce que vous avez modifié"

Exemples de messages de commit :

  • "Ajout du champ code postal dans le formulaire d'inscription"
  • "Correction affichage mobile du header"

Étape 2 — Déployer

npm run deploy

Cette commande :

  1. Compile le projet (npm run build)
  2. Publie automatiquement le résultat sur GitHub Pages

Attendre 1 à 2 minutes puis actualiser le site pour vérifier.

Étape 3 — Sauvegarder le code source sur GitHub (recommandé)

git push origin main
💡 Cette commande sauvegarde le code source sur GitHub en plus du site déployé. En cas de problème, vous pourrez toujours récupérer une version précédente.

6. Mises à jour Supabase

Supabase gère les comptes membres, les inscriptions et les profils.
Adresse : https://supabase.com → se connecter → projet AMSP

Modifier des données (Table Editor)

  1. Menu gauche → Table Editor
  2. Sélectionner la table (profils, inscriptions…)
  3. Cliquer sur une cellule pour la modifier directement
  4. Valider avec Entrée

Cas courants

Action Où
Changer le rôle d'un membre (membre / admin / admin_discipline) Faisable directement depuis le site → Gestion des membres
Voir les inscriptions Table inscriptions
Supprimer un compte Table profils → sélectionner la ligne → icône poubelle

disciplinestextDisciplines de l'adhérent (ex: Karaté, Judo). Utilisé pour filtrer la galerie privée et les droits admin_discipline.

⚠️ Si vous reconstruisez la base de données depuis zéro, pensez à créer toutes ces colonnes. Le script SQL complet est disponible dans l'historique git.

Modifier la structure des tables (SQL Editor)

  1. Menu gauche → SQL Editor
  2. Écrire la commande SQL, par exemple :
ALTER TABLE profils ADD COLUMN nouvelle_colonne text;
  1. Cliquer sur Run

Modifier les politiques d'accès (Row Level Security)

  1. Menu gauche → Authentication → Policies
  2. Sélectionner la table
  3. Modifier ou créer une politique

Rôles disponibles dans profils.role

ValeurAccès sur le site
adminSuper-admin : voit toutes les inscriptions, tous les membres, peut exporter, gérer les rôles
admin_disciplineAdmin limité : voit uniquement les inscriptions de sa discipline (colonne disciplines)
membreMembre approuvé : accès à la galerie privée de sa discipline
en_attenteDemande d'accès en cours de traitement
refuseDemande refusée

Politiques RLS requises

Les politiques suivantes doivent exister sur la table inscriptions pour que les admins puissent valider/refuser les dossiers. Si jamais les boutons Valider/Refuser semblent fonctionner mais que le statut revient en attente au refresh, c'est qu'une politique est manquante.

Exécuter dans le SQL Editor :

-- Permettre aux admins de modifier les inscriptions CREATE POLICY "Admins peuvent modifier les inscriptions" ON inscriptions FOR UPDATE USING ( EXISTS ( SELECT 1 FROM profils WHERE profils.id = auth.uid() AND profils.role = 'admin' ) ) WITH CHECK ( EXISTS ( SELECT 1 FROM profils WHERE profils.id = auth.uid() AND profils.role = 'admin' ) );
⚠️ Sans cette politique, Supabase bloque silencieusement les UPDATE (aucune erreur retournée, 0 lignes modifiées). L'interface semble fonctionner mais la base de données n'est pas mise à jour.

7. Brevo (emails automatiques)

Brevo envoie les 7 emails automatiques du site (inscriptions, validations, notifications admin) et sert de relais SMTP pour les emails d'authentification Supabase (réinitialisation de mot de passe, invitations membres).
Adresse : https://www.brevo.com → se connecter

Modifier un template

  1. Menu gauche → Email → Templates
  2. Cliquer sur le template à modifier → Modifier
  3. Modifier le contenu HTML
  4. Cliquer sur Enregistrer et quitter

Les variables dans les templates utilisent la syntaxe : {{ params.nom }}, {{ params.email }}, {{ params.disciplines }}…

Clé API et configuration

La configuration Brevo est répartie en deux endroits distincts :

VariableOù la configurerRôle
BREVO_API_KEY Supabase → Edge Functions → Secrets Clé API Brevo pour l'envoi des emails (côté serveur)
BREVO_SENDER_EMAIL Supabase → Edge Functions → Secrets Adresse expéditrice des emails (ex : contact@amsp.fr)
VITE_BREVO_ADMIN_EMAIL Fichier .env à la racine du projet Email destinataire de l'admin (notifications d'inscription)
VITE_BREVO_TEMPLATE_* Fichier .env à la racine du projet IDs des 7 templates Brevo (1 à 7)
💡 Pour modifier l'adresse expéditrice ou la clé API : Supabase Dashboard → Project → Edge Functions → Secrets → modifier BREVO_SENDER_EMAIL ou BREVO_API_KEY.
Pour modifier l'email de réception des notifications admin : modifier VITE_BREVO_ADMIN_EMAIL dans le fichier .env puis redéployer (npm run deploy).
⚠️ Le fichier .env n'est pas envoyé sur GitHub. Sur un nouvel ordinateur, copier .env.example en .env et remplir les valeurs. La clé API et l'email expéditeur sont à récupérer dans Supabase Secrets.

8. Contenu du site (Sanity CMS)

Le contenu éditorial (disciplines, actualités, galerie, textes du formulaire…) est géré dans Sanity Studio.

Lancer Sanity Studio en local

cd C:\Dev\AMSP\amsp-project\amsp-admin npm run dev

Ouvrir dans le navigateur : http://localhost:3333

Contenu modifiable

  • Disciplines : textes, images, descriptions — chaque discipline a sa page détail (/disciplines/:slug) avec instructeurs associés et liens externes configurés dans le code (LIENS_UTILES dans DisciplineDetail.tsx)
  • Actualités : articles, photos
  • Galerie : albums photos (privés ou publics) — les albums privés peuvent être associés à une discipline : seuls les membres de cette discipline les voient
  • Inscription : textes du formulaire (règlement intérieur, infos paiement, autorisation parentale…)
  • Instructeurs : fiches des instructeurs, avec champ liens pour leurs réseaux sociaux / sites externes
  • Paramètres : informations de contact, horaires
💡 Les modifications Sanity sont publiées immédiatement sur le site — pas besoin de redéployer.

Sauvegarder le contenu Sanity avant toute modification importante

Avant de modifier la saison ou tout autre paramètre critique, exporter l'intégralité du contenu Sanity :

  1. Créer un dossier de sauvegarde si besoin : C:\Dev\AMSP\sauvegarde\
  2. Ouvrir un terminal dans ce dossier et exécuter :
npx sanity@latest dataset export production backup-sanity.tar.gz --project-id t7u49x1d
  1. Se connecter avec le compte Sanity si demandé
  2. Le fichier backup-sanity.tar.gz est créé — le conserver précieusement
⚠️ La CLI Sanity nécessite Node.js 22 ou supérieur pour la version 6. Si votre version est plus ancienne, utilisez à la place l'URL d'export directe dans votre navigateur (connecté à sanity.io) :

https://t7u49x1d.api.sanity.io/v2021-06-07/data/export/production

Puis Ctrl+S pour sauvegarder le fichier téléchargé.

Restaurer un backup Sanity

En cas de problème, restaurer le contenu depuis le fichier de sauvegarde :

npx sanity@latest dataset import backup-sanity.tar.gz --dataset production --project-id t7u49x1d
⚠️ La restauration écrase le contenu actuel du dataset. À n'utiliser qu'en dernier recours.

Changer de saison

Pour passer à une nouvelle saison (ex. : de 2025-2026 à 2026-2027) :

  1. Sauvegarder le dataset Sanity (voir ci-dessus) avant toute manipulation
  2. Ouvrir Sanity Studio → Inscription
  3. Modifier le champ Saison avec la nouvelle valeur (ex. : 2026-2027)
  4. Publier le document
💡 Le champ Saison est une simple chaîne de texte. Toutes les nouvelles inscriptions reçues après ce changement auront automatiquement la nouvelle saison. Les anciennes inscriptions (saison précédente) restent intactes dans Supabase et restent accessibles en changeant le filtre Saison dans la page Gestion des membres.

Si vous revenez à une ancienne valeur de saison (ex. : pour corriger une erreur), les nouvelles inscriptions créées fusionneront naturellement avec celles déjà existantes pour cette saison — il n'y a pas de duplication.

10. En cas de problème

Problème Solution
npm run dev ne démarre pas Vérifier que Node.js est installé : taper node -v dans le terminal. Si erreur, réinstaller Node.js.
npm run deploy échoue Vérifier la connexion internet et les droits GitHub (être bien connecté dans VSCode).
Le site ne se met pas à jour Vider le cache navigateur : Ctrl + Maj + R, attendre 2 min. Si toujours pas à jour : Ctrl + Maj + Suppr → effacer le cache.
Erreur "permission denied" sur Git Vérifier que vous êtes bien connectée à GitHub dans VSCode (icône de compte en bas à gauche).
Les emails automatiques du site n'arrivent pas (inscription, validation…) Vérifier sur brevo.com → Transactionnel → Logs. Vérifier que BREVO_API_KEY et BREVO_SENDER_EMAIL sont bien renseignés dans Supabase → Edge Functions → Secrets. Vérifier que VITE_BREVO_ADMIN_EMAIL dans .env est l'adresse correcte.
L'email "mot de passe oublié" ou l'invitation membre n'arrive pas Ces emails transitent par Brevo SMTP (section 7). Vérifier dans Supabase → Authentication → Logs. Si erreur 500 : vérifier les identifiants SMTP dans Supabase → Project Settings → Auth → SMTP Settings (username = a9b934001@smtp-brevo.com, pas l'email de connexion Brevo). Si erreur 200 mais email absent : vérifier que la restriction IP Brevo est bien désactivée pour les clés SMTP (Brevo → Sécurité → IPs autorisées).
Le fichier .env est manquant Copier .env.example en .env et remplir VITE_BREVO_ADMIN_EMAIL et les IDs de templates. La clé API Brevo est dans Supabase Secrets, pas dans .env.
Les boutons Valider/Refuser semblent fonctionner mais le statut revient en attente Politique RLS manquante sur la table inscriptions. Exécuter le script SQL de la section 6 (Politiques RLS requises) dans le SQL Editor de Supabase.
